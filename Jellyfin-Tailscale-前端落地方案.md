# Jellyfin + Tailscale 私有视频分享与投屏方案

## 1. 文档目标

这份文档的目标不是直接“做一个视频网站”，而是先搭出一套可长期使用的私有视频方案：

- 视频文件仍然保存在你自己的电脑、NAS 或家庭服务器上
- 不把视频上传到公开视频平台
- 不把视频暴露到公网
- 允许你自己的手机、平板、电脑、电视等终端访问
- 重点保证“投屏”体验可用
- 前端后续可以在这套基础上继续做你自己的网页入口

这份文档也将作为后续实现方案的基线。

## 2. 先讲结论

如果你的核心需求是：

- 本地保存
- 不上公开云
- 多终端访问
- 投屏能力强
- 后续还想做自己的网页

那么 `Jellyfin + Tailscale` 是一条很稳的路线。

它的核心分工其实很简单：

- `Jellyfin` 负责“媒体服务”
- `Tailscale` 负责“私有网络连接”
- 你后续要开发的网页，负责“自己的前端入口、组织方式、学习场景”

也就是说：

1. 视频文件继续在你的本地机器上
2. Jellyfin 把这些本地文件变成“可播放、可管理、可跨端访问”的媒体库
3. Tailscale 让只有你授权的设备能连进来
4. 你的网页不需要直接裸读本地文件，而是调用 Jellyfin 的能力

从前端开发角度看，这个方案的价值在于：

- 你不用自己从零做视频转码、字幕、断点续播、媒体索引、用户鉴权
- 你不用自己处理公网暴露、动态 IP、端口映射这些网络问题
- 你可以先把“能看、能投、能分享给自己的设备”做通，再决定前端要不要深度定制

## 3. 为什么它能做到“本地保存、不上公开云”

这件事最容易误解的地方是：很多人以为“远程访问”就等于“先把文件传到云上”。

这里不是这样。

### 3.1 视频实际存在哪里

视频文件实际存放在你自己的设备上，例如：

- 一台常开电脑
- 一台 Mac mini
- 一台 Windows 主机
- 一台 Linux 小主机
- 一台 NAS

Jellyfin 只是扫描这些目录，把它们注册成媒体库。例如：

```text
/Volumes/Media/Courses
/Volumes/Media/Videos
/Users/you/Videos/PrivateLearning
```

Jellyfin 不会把这些文件“搬到 Jellyfin 云上”，因为 Jellyfin 本身就是自托管服务。

### 3.2 远端设备是怎么访问到这些视频的

远端设备不是去访问“公开网站”，而是通过 `Tailscale` 加入你的私有网络。

可以把它理解成：

- 你的电脑装了 Tailscale
- 你的手机也装了 Tailscale
- 它们登录到同一个 tailnet
- 于是它们像处在一个受控的私有局域网里

然后你的手机访问的不是公开地址，而是类似：

```text
http://100.x.x.x:8096
```

或者：

```text
http://your-server-name:8096
```

这里的 `100.x.x.x` 是 Tailscale 分配的私有地址，不是面向全网公开的公网地址。Tailscale 官方文档说明，tailnet 是一个私有空间，不对公网开放；设备之间通过加密点对点连接通信，且不需要你自己做端口映射。  
参考：

- <https://tailscale.com/docs/concepts/what-is-tailscale>
- <https://tailscale.com/docs/concepts/tailnet>
- <https://jellyfin.org/docs/general/post-install/networking/tailscale/>

## 4. 为什么它能做到“跨端”

从前端视角，跨端其实分两层：

- 一层是“同一份视频内容能不能在不同设备上被消费”
- 一层是“同一套交互入口能不能在不同终端上成立”

Jellyfin 先解决第一层。

### 4.1 Jellyfin 提供的是统一媒体服务，不是单一网页

Jellyfin 可以被多种客户端访问：

- 浏览器
- 手机 App
- 平板 App
- 电视端客户端
- 其他支持 Jellyfin 的播放器

这意味着你不是在给每个终端单独发视频文件，而是在给每个终端提供“访问同一个媒体服务”的能力。  
参考：<https://jellyfin.org/docs/general/clients/>

### 4.2 前端上真正统一的是什么

对于你未来自己开发的网页来说，真正统一的是：

- 认证方式
- 媒体列表结构
- 封面、标题、分类、标签
- 播放入口
- 观看记录
- 收藏、继续播放

也就是说，你后续自己的前端更像一个“学习视频门户”，而 Jellyfin 是底层媒体引擎。

## 5. 为什么它的“投屏”能力会比你自己从零做网页更靠谱

这是最关键的一部分。

### 5.1 很多人以为“网页里放一个 video 标签”就等于支持投屏

其实不是。

一个普通网页播放器，最多只是：

- 在浏览器里播放本地或远程视频
- 借助浏览器自身的一些投放能力做有限支持

但如果你要的是稳定的投屏体验，实际涉及的是：

- 设备发现
- 播放控制
- 媒体格式兼容
- 网络通路
- 字幕处理
- 码率适配
- 断点续播
- 遥控暂停/快进/切集

这已经不是“写一个前端页面”能轻松覆盖的范围了。

### 5.2 Jellyfin 的价值在于，它已经帮你处理了媒体服务层的大头

从前端开发角度，视频体系通常分成三层：

#### 第 1 层：文件层

- 你硬盘里的 mp4、mkv、srt

#### 第 2 层：媒体服务层

- 文件扫描
- 媒体库
- 元数据
- 转码
- 字幕
- 播放会话
- 用户权限

#### 第 3 层：前端展示层

- 首页
- 分类页
- 播放页
- 学习笔记页
- 投屏按钮

你如果直接自己做网页，相当于要从第 2 层和第 3 层一起做，成本非常高。

而 Jellyfin 已经把第 2 层做掉了，所以你后面只需要重点做第 3 层。

### 5.3 “投屏强”真正来自哪里

“投屏强”并不是说 Jellyfin 神奇地发明了投屏，而是它已经有成熟客户端和媒体服务配合来支撑这件事：

- 浏览器端可直接看
- 移动端更适合做投屏控制器
- 电视端可直接装客户端，很多场景甚至不需要传统意义上的“投屏”

这里要区分两个概念：

#### A. 真投屏

手机或网页把播放任务发给电视/投屏设备，电视自己播放。

#### B. 电视直连 Jellyfin

电视上直接装客户端，自己从 Jellyfin 拉流。

很多家庭场景里，`B` 比 `A` 更稳定，因为电视不是“被动接收”，而是“主动播放”。从使用体验上，用户仍然会觉得这是一种非常接近投屏的电视观看方式。

因此从产品设计角度，你后续前端不应该只押注“浏览器一键 Cast”，而应该同时支持：

- 手机选片 -> 电视端 Jellyfin 播放
- 手机选片 -> 系统投屏
- 电脑网页直接播放

## 6. 从前端开发角度，Jellyfin + Tailscale 的角色分工

你可以把整套系统理解成下面这张图：

```text
[你的硬盘视频]
       |
       v
[Jellyfin Server]
  - 扫描媒体库
  - 提供封面/标题/分类
  - 提供播放地址
  - 处理转码/字幕/续播
       |
       v
[Tailscale Tailnet]
  - 让授权设备进入私有网络
  - 不暴露公网端口
       |
       v
[你的前端网页 / 浏览器 / 手机 / 平板 / 电视]
```

对前端来说，实际会接触的是 Jellyfin 的接口和播放能力，而不是直接去访问本地磁盘文件。

### 6.1 前端将来主要做什么

你的网页更适合做这些事情：

- 登录入口
- 首页推荐
- 课程集合页
- 视频专题页
- 搜索
- 标签过滤
- 学习进度
- “继续观看”
- 自定义封面和文案
- 自定义分享页

### 6.2 前端不应该优先重造什么

前期不建议自己重造：

- 视频转码
- HLS 切片
- 播放权限体系
- 视频索引数据库
- 字幕烧录
- 投屏协议适配

这些更适合先交给 Jellyfin。

## 7. 一套通俗的使用流程

你可以把实际使用过程理解成下面这样：

### 场景 1：手机看你本地电脑里的视频

1. 你的电脑上放着视频文件
2. Jellyfin 扫描到这些视频
3. 电脑上运行 Tailscale
4. 手机上也运行 Tailscale
5. 手机加入同一个 tailnet
6. 手机打开 Jellyfin 地址
7. 手机看到媒体库，点击播放

这时视频并没有上传到公开平台，而是手机通过私有网络去访问你自己的媒体服务器。

### 场景 2：手机把视频投到电视

有两条路：

#### 路线 A：电视本身有 Jellyfin 客户端

1. 电视加入同一个网络环境
2. 电视端安装 Jellyfin 客户端
3. 电视直接连接你的 Jellyfin Server
4. 你用电视遥控器选片，或者用手机辅助控制

这是最稳的一条。

#### 路线 B：手机做控制器，投给电视

1. 手机打开 Jellyfin 或你的前端网页
2. 用户点击投屏
3. 投屏设备开始拉取或接收媒体流

这条路线在体验上更像“传统投屏”，但兼容性更依赖终端、协议和播放器能力。

所以产品策略上建议：

- 把“电视直接装 Jellyfin 客户端”当主路径
- 把“网页/手机投屏”当增强能力

## 8. 为什么 Tailscale 很关键

如果没有 Tailscale，你一般会遇到这些问题：

- 家里宽带没有固定公网 IP
- 路由器端口映射麻烦
- 某些运营商会拦截入站访问
- 外网访问要自己做 HTTPS、反代和安全控制

而 Tailscale 把问题改写成：

- 只允许加入 tailnet 的设备访问
- 设备之间通过加密通道通信
- 尽量走点对点连接
- 不需要先把 Jellyfin 暴露成公开网站

对前端开发的意义在于：

- 你做页面时，网络路径更稳定
- 你不用一开始就处理公网安全暴露问题
- 你可以先把产品流程打磨好

## 9. 从“我要做一个网页”出发，正确的落地顺序

这里最重要的是顺序。

如果一上来就开发完整网页，你很容易陷入下面这些坑：

- 页面做好了，但视频服务不稳定
- 页面能看，电视不能投
- 本地播放没问题，远程访问失败
- 链接能打开，但字幕、封面、进度都不统一

所以正确顺序不是：

1. 先写网页
2. 再想办法让它播放本地视频

而是：

1. 先把媒体服务跑通
2. 再把私有网络跑通
3. 再做前端入口
4. 最后再做投屏增强和体验优化

## 10. 建议的实施阶段

后续实现建议分四个阶段。

### 阶段 0：验证基础链路

目标是确认这条路线在你的设备环境里成立。

要完成的事情：

- 安装 Jellyfin Server
- 添加一个测试视频库
- 安装 Tailscale
- 用另一台设备通过 tailnet 访问 Jellyfin
- 验证浏览器播放
- 验证手机播放
- 验证电视端可行性

阶段 0 的交付标准：

- 至少 1 台服务器能跑 Jellyfin
- 至少 1 台手机能通过 Tailscale 访问
- 至少 1 个视频可正常播放

### 阶段 1：建立“自己的网页入口”

目标是先做一个轻量前端壳，不重造播放器体系。

建议这个阶段的网页只做：

- 首页
- 视频列表页
- 分类页
- 视频详情页
- 跳转/嵌入 Jellyfin 播放

阶段 1 不建议做的事情：

- 自己实现完整登录系统
- 自己重做媒体管理后台
- 自己写转码服务

### 阶段 2：前端与 Jellyfin 深度整合

目标是开始把“你的网页”做成真正的产品。

可选能力：

- 使用 Jellyfin API 拉取媒体列表
- 自定义首页推荐
- 自定义专题和课程编排
- 自定义继续观看
- 用户侧学习记录
- 收藏与标签系统

### 阶段 3：投屏与电视体验增强

目标是让大屏体验更像正式产品。

可选能力：

- 播放页投屏入口
- 为电视端设计更大的卡片和遥控器交互
- 强化播放态同步
- 优化字幕和清晰度切换
- 针对移动端设计“遥控器模式”

## 11. 面向前端的技术架构建议

后续如果要正式开发网页，建议采用下面这个思路。

### 11.1 架构原则

- Jellyfin 负责媒体服务
- 你的前端负责内容组织和体验
- Tailscale 负责私有访问链路
- 前期尽量少造轮子

### 11.2 推荐结构

```text
前端 Web App
  -> 调用你自己的 BFF（可选）
  -> 或直接调用 Jellyfin API

Jellyfin Server
  -> 管理视频文件
  -> 提供媒体元数据和播放能力

Tailscale
  -> 保障只有授权设备可访问
```

### 11.3 前端技术建议

如果后续要做网页，建议优先考虑：

- `Next.js` 或 `Vite + React`
- 播放器先用浏览器原生 `video`，后期再接更复杂播放器
- 首期重点做信息架构，不先碰复杂播放器协议

前端首期页面建议：

- 登录引导页
- 首页
- 视频列表页
- 视频详情页
- 播放页
- 设备连接说明页

## 12. 前端调研清单

后续开始实现前，需要先调研清楚下面这些问题。

### 12.1 设备与网络调研

- 服务器准备放在哪台设备上
- 这台设备是否能长期在线
- 家里/办公室网络是否稳定
- 手机、平板、电视分别是什么系统
- 电视能否安装 Jellyfin 客户端
- 哪些设备需要安装 Tailscale

### 12.2 媒体文件调研

- 视频格式主要是 mp4 还是 mkv
- 是否有外挂字幕
- 是否需要多音轨
- 视频分辨率和码率大概多少
- 服务器是否有足够 CPU 做转码

### 12.3 前端产品调研

- 你这个网页主要是给自己用，还是给少量固定对象用
- 首页是按课程、合集、主题，还是按文件夹组织
- 播放页是否需要笔记区
- 是否需要“继续学习”
- 是否需要公开分享页面

### 12.4 投屏调研

- 主要投到什么设备
- 是 Android TV、Apple TV、Chromecast、投影仪，还是国产电视
- 你要的是“电视装客户端”，还是“手机遥控投屏”
- 是否需要浏览器里直接显示投屏按钮

## 13. 可逐步操作的实施步骤

这一节是执行清单，后续可以按顺序落地。

### 第一步：确定服务器设备

建议先确定 1 台长期在线的机器作为 Jellyfin Server。

要求：

- 硬盘能放视频
- 能长期开机
- 局域网稳定

优先级建议：

1. NAS
2. 小主机 / Mac mini
3. 台式机
4. 笔记本

### 第二步：准备视频目录

先把视频文件整理好，按相对稳定的结构存放，例如：

```text
Media/
  Courses/
    Topic-A/
    Topic-B/
  Collections/
    Collection-1/
  Clips/
```

要求：

- 文件名尽量规范
- 封面和字幕单独管理
- 后续不要频繁大改目录结构

### 第三步：安装并启动 Jellyfin

在服务器设备上安装 Jellyfin。

完成后要做：

- 进入 Jellyfin 后台
- 创建管理员账号
- 添加视频库目录
- 确认媒体已被扫描

验收标准：

- 浏览器在服务器本机可以打开 Jellyfin
- 能看到至少 1 个视频条目
- 点击后可以播放

### 第四步：安装并配置 Tailscale

在以下设备上安装 Tailscale：

- Jellyfin 服务器
- 你的手机
- 你的电脑
- 其他需要访问的终端

完成后要做：

- 所有设备登录同一个 tailnet
- 记下 Jellyfin 服务器的 Tailscale IP
- 在另一台设备上用该地址访问 Jellyfin

例如：

```text
http://100.x.x.x:8096
```

验收标准：

- 非服务器本机也可以访问 Jellyfin
- 不需要路由器端口映射
- 外部网络下仍然可访问

### 第五步：验证最小可用播放链路

至少做三项验证：

1. 电脑浏览器播放
2. 手机播放
3. 电视端播放或电视端客户端连接

如果第 3 项不稳定，优先排查：

- 电视系统兼容性
- 电视是否适合直接装 Jellyfin 客户端
- 是否需要改成手机控制 + 电视播放的路径

### 第六步：定义你的网页第一版范围

第一版网页建议非常克制，只做下面几个页面：

- 首页
- 分类页
- 视频详情页
- 播放入口页

并且采用下面这个原则：

- 页面负责“找内容”和“理解内容”
- Jellyfin 负责“实际播放能力”

### 第七步：开始前端开发

前端第一版建议目标：

- 统一视觉
- 更适合你的使用习惯
- 更像“学习内容门户”

而不是：

- 立刻替换掉 Jellyfin 全部能力

### 第八步：决定和 Jellyfin 的集成深度

你有两条路线可以选。

#### 路线 1：轻集成

你的网页只做：

- 首页与分类
- 视频说明
- 链接跳转 Jellyfin 播放页

优点：

- 开发快
- 风险低
- 先把体验搭起来

缺点：

- 风格不完全统一

#### 路线 2：深集成

你的网页开始直接读取 Jellyfin API，并自己管理播放页和部分状态。

优点：

- 体验更统一
- 定制能力更强

缺点：

- 前端和媒体系统耦合更深
- 调试成本更高

建议顺序：

先做路线 1，再逐步过渡到路线 2。

## 15. 投屏功能的产品建议

如果你后续一定要把“投屏”做成网页里的明确功能，建议按下面这个优先级做。

### 优先级 1：电视端直接连接 Jellyfin

这是最稳、最不折腾、体验最容易闭环的一种方式。

你的网页里只需要明确告诉用户：

- 推荐在哪个电视端安装客户端
- 如何连接 Jellyfin 服务器
- 如何在电视上继续播放

### 优先级 2：移动端作为控制入口

网页或手机端提供：

- 选择视频
- 打开到播放器
- 显示“去电视上播放”的引导

### 优先级 3：网页内增强投屏

只有在前两者都跑稳之后，再考虑：

- 更细致的投屏按钮
- 播放状态同步
- 设备发现体验

## 16. 风险与边界

后续实现时要提前接受几个现实。

### 16.1 不是所有“投屏”都能只靠网页完成

浏览器前端对不同投屏协议的控制能力并不统一，设备兼容性也存在差异。

所以不能把产品成败只押在“网页直接一键投屏到所有电视”这件事上。

### 16.2 电视客户端往往比网页投屏更稳定

如果目标是“我想在电视上看”，那电视直接装客户端通常是更现实的主路径。

### 16.3 转码能力影响体验

如果源视频格式复杂，而服务器性能不足，远端播放和大屏播放都会受影响。

### 16.4 私有分享不等于无限制传播

这套方案适合私有、受控、授权范围内的学习访问，不适合把你没有分发权限的内容对外公开传播。后续产品设计上仍然应该把访问控制做好。

## 17. 当前推荐的实现策略

如果现在就开始落地，我建议采用这个策略：

### 17.1 近期策略

- 先不自己重写播放器
- 先不自己重做投屏协议
- 先用 Jellyfin 跑通跨端与电视访问
- 先用 Tailscale 跑通私有远程访问

### 17.2 中期策略

- 开发一个自定义前端门户
- 让它承载你的内容组织方式和学习体验
- 与 Jellyfin 做轻量集成

### 17.3 后期策略

- 再决定是否做深度 API 集成
- 再决定是否做更复杂的播放页和投屏控制
